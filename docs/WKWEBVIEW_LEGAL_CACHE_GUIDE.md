# WKWebView 法務ページ表示＋オフラインキャッシュ実装ガイド（iOS汎用）

このガイドは、利用規約・プライバシーポリシーなどの外部Webページをアプリ内表示しつつ、通信失敗時にキャッシュ表示するための実装ノウハウをまとめたものです。

## 目的
- 通常時は最新の法務ページをWebから表示する
- オフライン時や一時障害時でも、前回取得済みの内容を表示する
- 審査/運用で問題になりやすい白画面や壊れた表示を避ける

## 推奨アーキテクチャ
1. 表示は `WKWebView.load(URLRequest)` を第一優先
2. 表示成功時にHTMLを保存（`document.documentElement.outerHTML`）
3. 読み込み失敗時のみ `loadHTMLString` でキャッシュ表示
4. キャッシュがない場合はエラー用HTMLを表示

表示優先順位:
- `Remote URL` > `Cached HTML` > `Fallback Error HTML`

## 失敗しやすいポイントと対策
### 1) 常時 `loadHTMLString` を使う
- 問題: 相対パスのCSS/JSが壊れ、白画面化しやすい
- 対策: 通常表示は必ず `URLRequest` で読み込む。`loadHTMLString` はフォールバック専用

### 2) DOM非表示スクリプトの対象が広すぎる
- 問題: 不要リンクを消すつもりで親要素まで消して本文が消える
- 対策: セレクタは最小範囲（対象要素そのもの）に限定する

### 3) 失敗HTMLを上書きキャッシュしてしまう
- 問題: 次回以降も壊れた表示が再利用される
- 対策: 保存は「Remote表示成功時のみ」

### 4) フォールバック判定が早すぎる
- 問題: 一時的な読み込み遅延で不要にキャッシュへ切替
- 対策: `didFail` / `didFailProvisionalNavigation` の確定失敗でのみフォールバック

### 5) キャッシュ格納先が不安定
- 問題: 一時領域や権限不整合で消失
- 対策: `Application Support/<任意ディレクトリ>` を使用し、起動時に作成

## 実装チェックリスト
- [ ] Remote URL の表示経路とキャッシュ経路を分離した
- [ ] キャッシュ保存は成功時のみ
- [ ] ネットワーク失敗時にのみキャッシュ表示
- [ ] キャッシュファイルは `Application Support` 配下
- [ ] 白画面時の最終フォールバック（エラーHTML）を用意
- [ ] 「今何を表示しているか」をログ/状態表示で確認できる

## 監視・デバッグ推奨
- ログ例:
  - `legal_remote_load_started`
  - `legal_remote_load_succeeded`
  - `legal_cache_saved`
  - `legal_fallback_to_cache`
  - `legal_fallback_error_html`
- QA確認:
  - オンライン初回表示
  - オフライン表示（キャッシュ有）
  - オフライン表示（キャッシュ無）
  - 外部ページのHTML変更後の再取得

## ロールバック方針
- 不具合時はキャッシュ機能のみ一時OFFにして Remote表示のみに戻す
- その間、オフライン時は明示的な「通信が必要です」画面を表示
- 復旧後に再度キャッシュ機能を有効化

## 運用ルール（複数アプリ共通）
- 法務URLは設定値化（環境差分対応）
- キャッシュTTL（例: 7日）を必要に応じて設計
- 文言非表示用スクリプトは変更影響が大きいため、対象DOM変更時に回帰テストを必須化
